Complete Brand & Game Design Brief

Prompt Quest.

An AI-powered game teaching kids (ages 8-10) prompt engineering through play. Kids describe solutions, Claude brings them to life, and every mistake is funnier than getting it right.

Brand Story

The origin, the metaphor, the soul of the game
The Name

Prompt Quest — an adventure where your words are your superpower. Every quest begins with a blank text box and a wild scenario. What you type determines what happens. Say something specific and creative? Magic unfolds. Say something vague? Hilarious chaos. The "quest" isn't to find the right answer — it's to discover that how you ask changes everything. Kids don't know they're learning prompt engineering. They think they're playing the funniest game ever made.

The Theater

Imagine a magical theater where the audience writes the script. The stage has actors, props, and backdrops — all ready to go. But nothing happens until someone in the audience shouts out what should happen next. The more specific and creative the direction, the more amazing the show. Vague directions? The actors improvise — and it's hilarious. That's Prompt Quest. Claude is the director. Phaser is the stage. The kid is the playwright.

The Magic

The real magic isn't the AI — it's the gap between what a kid imagines and what appears on screen. When a child types "launch the cat from a cannon into the birthday cake" and the game actually tries to show that, the gap between expectation and result is comedy gold. Success feels earned. Failure feels funny. And every attempt teaches the same lesson: specific words create specific results.

The Audience

Primary: Kids ages 8-10 (Piaget's Concrete Operational stage). They can think logically about concrete things, understand cause-and-effect, and are developing metacognition — but they still struggle with pure abstraction. They need to see the results of their words, not be told about "prompt engineering principles."

Secondary: Parents and educators who want their kids to develop AI literacy through play. The hackathon judges evaluating Opus 4.6 projects. Anyone who wants to see a kid type "give the monster a hat made of pizza" and watch the game actually do it.

Positioning

What Prompt Quest is — and what it absolutely isn't
Positioning Statement

For kids who will grow up working alongside AI, Prompt Quest is the game that teaches them how — not through lessons or chatbots, but through play. Describe a solution, watch it come alive, learn that specificity is a superpower. AI is invisible. Fun is the interface.

Prompt Quest Is
●A game first, educational tool second — kids never feel "taught"
●AI as invisible engine — Claude powers it, kids never see a chatbot
●Funny fails that teach — mistakes are entertaining, not punishing
●Concrete feedback — "You forgot to describe the walls!" not "Be more specific"
●Visually immediate — type words, see animations, feel the result
●Outrageously silly — monsters, robots, dinosaurs, octopuses
●Instant and reliable — cached responses, fallback scripts, never breaks
Prompt Quest Is Not
●ChatGPT with a cute skin — no chat interface, no conversational AI
●A coding tutorial — no code, no syntax, no IDE
●An AI assistant for kids — the AI never talks directly to the child
●Lecturing or preachy — no "good job!" walls of text
●Punishing for mistakes — failure is always funny, never frustrating
●Baby-ish — no baby colors, no patronizing tone, no "for little kids" aesthetic
●Complex or overwhelming — one task, one input, one magical result

Color System

The Prompt Quest palette — magical, bold, never baby-ish
Palette Philosophy

Prompt Quest's palette is built for dark, magical environments that feel like a game, not a classroom. The deep purple foundation creates a sense of nighttime adventure and mystery — like sneaking into a wizard's laboratory. Bright orange accents punch through the darkness as the action color — interactive, warm, inviting. Green signals success and rewards. Yellow signals "funny chaos." The palette avoids two traps: (1) bright primary colors that read as "for babies" to an 8-year-old, and (2) muted pastels that feel boring. Research from Lullabot and NN/Group shows kids ages 8-10 prefer rich, saturated palettes over both primary colors and muted tones. Think Minecraft meets Gravity Falls — colorful but with depth.

Color Rules
WCAG AA on dark backgrounds
All text colors must pass 4.5:1 contrast ratio against #1A0F2E. White text (#FFF) on purple backgrounds. Never use dim purple text on purple backgrounds.
Never use red for failure
Red triggers anxiety in kids. Use yellow/orange for "funny fail" states. Reserve soft red only for system-level warnings (connection lost, etc.), never for game outcomes.
Glow effects, not flat fills
Interactive elements should have subtle glow (box-shadow with 0.15 alpha of the element's accent color). This creates a magical, luminous feeling. Not neon — think fireflies.
3 colors max per screen
The dark background + one primary accent + one secondary accent per screen. Too many colors creates visual chaos. Let the game scene be the color explosion — the UI stays calm.

Typography

Kid-readable, game-appropriate, never condescending
Display / Headlines
Monster Birthday Party!

Nunito Extra Bold / Black (700-900) — rounded sans-serif that's friendly without being childish. The rounded terminals feel approachable for kids while the clean geometry keeps it from looking "baby-ish." An 8-year-old sees this as "game font," not "little kid font."

Sizes: Task titles: 28-34px. Section headers: 22-26px. Button labels: 16-18px. All bold (700+).
Body / UI Text
The monster catches the giant cake and does a happy dance! Everyone cheers as confetti fills the room. Next time, try describing what kind of music to play at the party!

Nunito Regular / SemiBold (400-600) — same family for body text maintains visual cohesion. At 16px+, Nunito is highly readable for developing readers. The x-height is generous, letterforms are open and unambiguous.

Minimum body: 16px. Narration text: 18-20px. Feedback tips: 14-16px. Never below 14px for any kid-facing text.
Typography Rules for Kids
Minimum 16px body text
Research shows 8-10 year olds need larger text than adults. 16px minimum for any readable content. 14px absolute floor for secondary labels.
Line height: 1.6-1.8
Kids' eyes need more vertical space between lines. Tight line-height causes reading fatigue and line-skipping. 1.65 for body, 1.15 for headlines.
Short lines: 45-65 characters
Kids lose their place on long lines. Keep narration text in a narrow column. Max-width: 540px for text blocks.
Weight over size for hierarchy
Use 3-4 font sizes max. Vary weight (400, 600, 700, 900) and color to create hierarchy. Too many sizes = visual noise.
No ALL CAPS for content
Kids ages 8-10 are still building reading fluency. ALL CAPS slows reading speed by 10-15%. OK for tiny labels and badges, never for sentences.
Reading level: Grade 3-4
All kid-facing text at Flesch-Kincaid grade 3-4. Short sentences. Common words. "The monster loved the cake!" not "The creature expressed satisfaction."
Font Stack
Primary: 'Nunito', -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif
Monospace (code/data): 'Space Grotesk', 'SF Mono', monospace
System fallback: system-ui, sans-serif

Voice & Tone

How Prompt Quest talks to kids, parents, and judges
The Prompt Quest Voice Test

Would a funny, encouraging older sibling say this? If it sounds like a teacher, a textbook, or a children's TV host — rewrite it. If it sounds like the coolest babysitter who makes everything an adventure — ship it.

Four Voice Pillars
Playfully Encouraging
Every attempt is celebrated, not judged. "Whoa, the monster just sneezed fire on the cake! Maybe next time tell it to blow out the candles gently?"
Genuinely Funny
Humor through absurd situations and surprising outcomes, never through mockery. The game laughs WITH the kid, never AT them. Funny fails > boring successes.
Concretely Helpful
Feedback is about the game, never abstract prompting theory. "You forgot to tell the robot how to cross the river!" — not "Try to cover all aspects of the problem."
Never Condescending
8-year-olds can smell when something is "for little kids." No baby talk, no over-simplified language, no "Great job, buddy!" Instead: "That was epic! What if you tried..."
Narrating a Success
✓ Prompt Quest says
"The monster catches the giant cake and does a happy dance! Confetti everywhere!"
✗ Prompt Quest never says
"Wonderful work! You successfully completed the task by providing detailed instructions. You should feel proud of your achievement!"
Narrating a Funny Fail
✓ Prompt Quest says
"The robot walks straight into the river. Pizza's soggy. Robot starts doing the robot dance. Not ideal."
✗ Prompt Quest never says
"Oops! That didn't quite work out. Don't worry though, mistakes are how we learn! Would you like to try again?"
Giving Feedback / Tips
✓ Prompt Quest says
"The robot didn't know how to cross the river! Next time, try telling it what to use — a bridge? A jetpack? A really long pizza?"
✗ Prompt Quest never says
"Remember, good prompts should address all parts of the problem. Try to think about potential obstacles and include solutions for them."
Task Introduction
✓ Prompt Quest says
"A monster is having its first-ever birthday party. It's 30 feet tall, breathes fire, and has never seen a cake before. What could possibly go wrong?"
✗ Prompt Quest never says
"Welcome to the Monster Birthday Party challenge! In this exercise, you will practice writing detailed prompts by planning a party for a monster character."
Writing Rules
●Narrations: max 20 words, present tense, one exclamation mark max
●Feedback tips: max 2 sentences, always game-specific ("the monster" not "your prompt")
●Use "you" rarely — describe what happens, not what the kid did
●One idea per sentence, active voice, concrete nouns
●Never say "Great job!" "Well done!" "Good try!" — these are empty calories
●Never use the word "prompt" in kid-facing text — say "description" or "plan"
●Never use educational jargon: "skills," "learning," "practice," "challenge"
●Never use scary words: "wrong," "failed," "error," "mistake"

Visual Language

The look and feel of every pixel on screen
Design Philosophy

Prompt Quest's visual language sits in a specific sweet spot: more polished than indie pixel art, less corporate than educational software. Think of the aesthetic as "Cartoon Network show rendered as a game UI." Rounded corners but not bubbly. Colorful but with a dark, cozy foundation. Playful icons but clean layout. The game canvas is where the chaos lives — the surrounding UI is calm, dark, and simple so the game scene can explode with color and motion without competing with the interface.

Visual Hierarchy
1. The Game Canvas (Hero)
70% of visual attention. The Phaser scene is the star — where animations play, where actors move, where the magic happens. 1024x576px, prominent, center stage.
2. The Input Area (Action)
20% of attention. The text box where kids type their plan. Big, inviting, with clear placeholder text. This is the "do something" zone.
3. Feedback Panel (Learn)
10% of attention. Narration text and concrete tips. Appears after the animation plays. Short, colorful, disappears when the kid starts typing again.
4. Chrome (Invisible)
Navigation, task selector, settings — should feel invisible until needed. Dark, minimal, no decoration. The less the kid notices the UI, the more immersed they are.
Component Style Guide
Border radius: 14-20px
Rounded enough to feel friendly, not so rounded that it looks like a toy. Buttons: 14px. Cards: 20px. Input fields: 14px. Avoid perfect circles on rectangular elements.
Elevation through glow, not shadow
Drop shadows look corporate. Instead, use a subtle colored glow: box-shadow: 0 0 20px rgba(accent, 0.1). Creates a magical, luminous depth.
Touch targets: 48px minimum
Kids' motor skills are still developing. Buttons: 48px height minimum, 56px preferred. Gaps between interactive elements: 12px minimum.
One primary action per screen
Kids get overwhelmed by choices. Each screen has ONE obvious thing to do: type, submit, select a task. Everything else is secondary and visually recessed.
Visual Anti-Patterns
No primary colors (red/blue/yellow) as fills
Primary color palettes read as "ages 3-5" to an 8-year-old. They'll literally say "this is for babies." Use rich, saturated, slightly sophisticated colors.
No clip-art or stock illustration style
Flat vector "corporate illustration" is what they see on school websites. Game assets should look like game assets — pixel art, sprite art, or hand-drawn cartoon style.
No loading spinners with "Please wait"
Kids won't wait. Any loading state should be entertaining — a funny loading message, an animation, something that keeps their eyes on the screen.
No walls of text
If a kid sees a paragraph, they skip it. Max 2 sentences in any visible text block. Use visuals, icons, and animations to communicate instead.

Game Feel

Animation timing, easing, feedback loops — the invisible magic that makes it feel alive
The Juice Principle

"Juice" is game design shorthand for maximum response to minimum input. When a kid presses the submit button, the entire screen should react — the button squishes, the text flies away, the game canvas glows, a loading animation spins, and then the scene EXPLODES into action. Every interaction should feel like dropping a stone into a pond. Ripples everywhere. Silence is death in a kids game — if something feels unresponsive for even 300ms, the kid thinks it's broken.

The 200ms Rule

Every input must produce visible feedback within 200ms. If the API call takes 3 seconds, the button must react in 100ms (squish + loading state). If the scene script takes 500ms to parse, the canvas must show a "thinking" animation immediately. Kids don't have the patience or trust to wait without feedback. Dead space = "it's broken" in their minds. Chain your feedback: button press (100ms) → loading animation (immediate) → thinking text (200ms) → scene plays (when ready).

Sound Design

Audio that enhances without overwhelming — the invisible 50% of game feel
Sound Philosophy

Sound in a kids game serves three purposes: feedback (confirming actions), emotion (building joy/surprise/anticipation), and atmosphere (creating a world). Every sound should feel like it belongs in a Pixar short — expressive, warm, slightly cartoony. Never shrill, never loud, never startling. Think "boing" not "BANG." Parents are in the room. The sound design should make them smile, not reach for the mute button.

Background Music

Style: Playful, looping, low-energy ambient. Think Animal Crossing meets lo-fi study beats. Ukulele, marimba, soft synth pads, pizzicato strings. 90-110 BPM — upbeat but not frantic.

Volume: Background music at 30-40% of max. Must never compete with SFX or narration. Auto-duck during scene playback (fade to 20%). Fade out completely during text input focus.

Variation: Different track per task theme. Monster Party = quirky circus. Robot Pizza = electronic funk. Dinosaur School = jaunty adventure.

Sound Effects Categories
👈
UI Clicks
Soft "pop" or "bloop" — never harsh clicks. Different pitch for different buttons.
✍
Typing
Optional soft keystroke sound (can be toggled off). Light, percussive, non-repetitive.
🚀
Submit
Satisfying "whoosh" + sparkle on submit. This is the "launch" moment — should feel powerful.
🎉
Success
Ascending chime + confetti burst. Triumphant but brief (1-2s). Think Mario coin x3.
😂
Funny Fail
Descending trombone "wah-wah" or cartoon "boing." Funny, not sad. Should make kid laugh.
💥
Scene Events
Splashes, boings, thuds, whooshes, pops — each scene action gets a cartoony SFX layer.
Audio Rules
●Always provide a visible mute button — some kids play in shared spaces
●Default music volume: 35%. Default SFX volume: 70%. Separate sliders.
●No sudden loud sounds. Peak volume across all SFX should be normalized.
●Sound must be optional — game must be fully playable with audio muted.
●No voice acting for narration — typewriter text is more flexible and cheaper to iterate
●No alarm/siren sounds — can trigger anxiety in neurodivergent kids
●No looping SFX without timeout — stuck sounds are a top kid UX complaint
●No audio autoplay on page load — wait for first user interaction (also a browser requirement)

Emotional Design

How every moment should make a kid feel — the real product
The Emotional North Star

"I want to try something weird and see what happens."

That sentence is the entire emotional design goal. If a kid feels safe enough to type something silly, curious enough to wonder what will happen, and delighted enough by the result to try again — you've won. Every design decision should be measured against this: does this make them more or less likely to try something weird?

🚀
Before Submit
Curious + Excited. "What will happen if I type THIS?" The input area should feel like a launchpad, not a test form.
✨
During Animation
Surprise + Anticipation. "Wait... is it actually going to...?" The scene should build slowly enough for anticipation to mount.
😂
After Funny Fail
Laughter + "I wanna try again!" The fail should be SO funny that the kid shows someone else. Comedy > correctness.
🎉
After Full Success
Pride + "Look what I did!" Confetti, cheering, the works. This is their creation — make them feel like a genius.
💡
Reading Feedback
"Oh! I should have..." The tip should spark an idea, not shame. It's a clue in a mystery, not a grade on a test.
💫
Trying Again
Confidence + Purpose. "This time I'll tell it about the bridge AND the jetpack." The retry should feel like a power-up, not a do-over.
Failure as Comedy — The Core Design Pattern

The most important emotional design decision in Prompt Quest: failure must be funnier than success. This is counterintuitive — most educational games try to make success exciting and failure neutral. Prompt Quest inverts this. When a kid's vague description causes the robot to walk into a river and start doing the robot dance, that's PEAK entertainment. The laughter dissolves any frustration. The kid WANTS to see what other funny things can happen. And in trying to make something specific happen, they naturally learn to write better descriptions. The pedagogy is embedded in the comedy.

✓ Emotional Do's
●Make funny fails genuinely hilarious — worth showing a friend
●Celebrate every attempt, not just successes
●Let kids replay the same task differently — repetition is how they learn
●Show progress: "2 out of 3 party problems solved!"
●Surprise them — unexpected results build curiosity
●Let parents watch — design for an audience of 2+
✗ Emotional Don'ts
●Never show a red "X" or failure icon — use yellow/orange with comedy
●Never compare to other kids ("Your friends scored higher!")
●Never show timers or countdown pressure
●Never block progress — every outcome unlocks the next attempt
●Never use the word "wrong" — there is no wrong, only hilarious
●Never require reading to understand what happened — animations tell the story

Backdrop Art Style

Illustrated, not photorealistic. Consistent across all scenes. Backdrops should look like storybook illustrations — warm lighting, slightly exaggerated perspective, rich colors. Think "children's book painted by a concept artist." All backdrops from the same source/artist to maintain visual consistency. Kenney.nl packs or itch.io art packs work well when chosen from a single cohesive set.

Character Art Style

Consistent scale, consistent style, consistent personality. All characters (actors) should come from the same art pack or be drawn in the same style. Mixing pixel art with vector art with hand-drawn looks like a ransom note. Characters should be:

●Expressive: big eyes, exaggerated features, readable emotions at small sizes
●Scaled consistently: all actors roughly same height on the stage (except "giant" variants)
●Multiple animation states: idle, happy, sad, action — minimum 4 per character
●Silhouette-readable: identifiable even as a small thumbnail
Placeholder System

Missing textures never crash the game. When SceneScriptPlayer encounters an asset that isn't loaded, it creates a colored rectangle with a text label instead. This is a development-only system — all placeholders should be replaced before demo. Placeholder colors: actors = purple rectangles, props = orange rectangles, backdrops = gradient fill.

Accessibility

Every kid deserves to play — no exceptions
Accessibility is Not Optional

10-15% of kids have some form of disability — learning differences, color vision deficiency, motor challenges, or attention difficulties. If the game only works for neurotypical kids with perfect vision and fine motor skills, it's broken for 1 in 7 players. These aren't edge cases — they're kids in every classroom.

Visual Accessibility
●Color contrast: WCAG AA (4.5:1) for all text. AAA (7:1) for body text on dark backgrounds.
●Color is not the only signal: Success/failure also indicated by icons, animations, and text — not just green/yellow.
●No color-only meaning: 8% of boys have color vision deficiency. Use shapes + labels alongside color.
●No flashing: No elements flash more than 3x/second (photosensitive seizure risk).
●Reduce motion option: Respect prefers-reduced-motion. Simplify animations, keep game playable.
Motor & Input Accessibility
●Touch targets: 48px minimum with 12px gaps between interactive elements.
●Keyboard navigation: Tab through all interactive elements. Enter to submit. Escape to close panels.
●No precision required: No drag-and-drop, no small click targets, no hover-only interactions.
●No time pressure: No timers, no countdown, no "type faster" mechanics.
●Forgiving input: Spelling doesn't matter. Claude handles misspellings gracefully.
Cognitive Accessibility
●One task at a time: Never show multiple things to do simultaneously. Sequential, clear flow.
●Reading level: Grade 3-4 (Flesch-Kincaid). Short sentences, common words, no jargon.
●Consistent layout: Same UI structure on every screen. Input always in the same place.
●Clear next action: One obvious button to press at any given moment.
●Unlimited attempts: No lives, no game over, no penalty for trying many times.
●No memory demands: Task requirements visible on screen, not just mentioned once.
Accessibility Checklist
☑ All interactive elements have visible focus indicators (2px purple outline)
☑ All images have alt text (or role="presentation" for decorative)
☑ Game narration displayed as text (not audio-only)
☑ Animations can be paused/reduced
☑ Color contrast passes WCAG AA minimum
☑ No keyboard traps (can always Tab away from any element)
☑ Audio controls visible and reachable
☑ Works without mouse (keyboard-only path through entire game)
