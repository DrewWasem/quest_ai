Prompt Quest ¬∑ Redesign ¬∑ Free-Play Architecture

From Scripted Scenarios to Sandbox Play

Kill the "guess what the designer wanted" scenarios. Replace them with a free-play sandbox where kids learn prompt literacy by experimenting, failing funny, and iterating ‚Äî the way real prompt engineering actually works. Practical, hackathon-scoped, buildable.

The Problem

Why the Scenarios Don't Work

‚ùå Current: Hidden Answer Design

"Nobody knows what to do! What game should they play?"

Expected answer: Musical chairs. But the child doesn't know chairs exist in the scene, doesn't know Walking_A ‚Üí Running_A ‚Üí Sit_Chair_Down is a chain, and has no reason to pick musical chairs over tag, hide and seek, or pin the tail on the donkey.

This teaches kids to guess the adult's mind. That's the opposite of prompt engineering.

‚úÖ New: Open Sandbox Design

"It's party time! Tell the characters what to do!"

Kid types "play tag" ‚Üí Claude maps it to Running_A + Dodge animations. Kid types "dance battle" ‚Üí Claude uses Dance_A + Dance_B. Kid types "throw cake" ‚Üí Throw animation + cake prop + splat effect. Every input produces something. Nothing is wrong.

This teaches kids that specificity gets better results. That's actual prompt engineering.

Core Insight: Real prompt literacy isn't about knowing the right answer. It's about learning that vague inputs ‚Üí weird outputs, and specific inputs ‚Üí better outputs. The learning loop is: try ‚Üí see result ‚Üí notice what's missing ‚Üí try again with more detail.
The New Model

Worlds, Not Scenarios

Replace the 7 rigid scenarios with 5 themed Worlds. Each World is a sandbox with a setting, a cast of characters, and a pile of props. The child gets a loose story hook, then drives the action through free-form prompts. There are no right answers ‚Äî just a spectrum from vague (funny, chaotic) to specific (impressive, controlled).

Learning Design

How Prompt Literacy Emerges from Free Play

The Natural Learning Loop

1
Kid types something vague

"make them do stuff"

2
System produces a funny but underwhelming result

All characters do Idle animation. Narrator: "They're... standing there! Doing stuff! Very... stuff-like. Maybe tell them WHAT stuff?"

3
Kid tries again with more detail

"make the skeleton dance"

4
System produces a better result

Skeleton does Dance_A. Narrator: "Now we're talking! The Skeleton's got moves! But the others look bored. What about them?"

5
Kid gets ambitious

"the skeleton dances while the knight watches and claps and then the robot does a backflip"

6
System produces something impressive

Skeleton ‚Üí Dance_A, Knight ‚Üí Idle_Clap (or closest), Robot ‚Üí Jump_Full_Long. Sequenced with timing. Narrator: "A SHOW! The crowd goes wild! That was a really specific prompt ‚Äî nice work!"

Gentle Scaffolding (Not Correct Answers)

Instead of pass/fail scoring, the system uses a Guide Character (a friendly narrator voice) who drops contextual hints based on what the child's prompt was missing. These aren't corrections ‚Äî they're suggestions that model the thinking process.

Progress System: Prompt Badges (Not Stars)

Instead of 1-3 stars per scenario, kids earn Prompt Badges that track the prompt engineering skills they've demonstrated across any world. These persist across sessions.

BadgeSkillHow to Earn
üéØ CommanderNamed a specific characterUse a character name in a prompt
üé¨ DirectorGave a specific actionUse an action verb (dance, run, fight, cook...)
üìù StorytellerWrote a multi-step sequenceUse "then" or "and then" or "first...next"
üé® Detail MasterAdded descriptive detailInclude adjectives or manner (slowly, angrily, twice)
üåç World BuilderReferenced the environmentMention a prop or location in the scene
üé™ Combo KingCoordinated multiple charactersGive instructions to 2+ characters in one prompt
üîÑ IteratoooorImproved on a previous promptRetry with more detail after a funny fail
‚≠ê Prompt MasterUsed 4+ skills in one promptCombine naming, action, sequence, and detail
Why badges > stars: Stars punish ("you only got 1 star"). Badges reward ("you just unlocked Director!"). A kid who types "dance" earns Commander even though it was vague ‚Äî because they DID name an action. Progress is always forward. This is how games like Minecraft and Roblox keep kids engaged without fail states.
Architecture

Technical Implementation (Hackathon-Scoped)

What Stays the Same

The core architecture from the implementation plan is solid and unchanged:

  • React + Three.js + Vite frontend
  • Claude Opus 4.6 as the "Assembler, Never Renderer" ‚Äî generates SceneScript JSON
  • SceneScriptPlayer sequences spawn/animate/move/react/sound/camera actions
  • GLTFLoader + AnimationMixer for all 3D
  • Vocabulary Contract constraining Claude to real assets

What Changes

Remove
  • 7 rigid task definitions with expected answers
  • Golden Response Cache (no "right" answers to cache)
  • Star rating (1-3) per scenario
  • Task-specific system prompts with hidden success criteria
  • Levenshtein fuzzy-matching to predetermined responses
Add
  • 5 World definitions (setting, characters, props, hook)
  • Prompt Analyzer (lightweight ‚Äî detects skills used in prompt)
  • Badge system with persistent progress
  • Guide character narration that adapts to what's missing
  • World-aware system prompt (tells Claude what's available)

Prompt Analyzer: Client-Side Badge Detection

The prompt_analysis from Claude drives badge unlocks on the client. This is lightweight ‚Äî no separate model call needed. Claude already understands the prompt; we just ask it to tag what skills it detected.

// Badge check runs after every SceneScript response function checkBadges(analysis: PromptAnalysis, badges: BadgeState): BadgeUnlock[] { const unlocks: BadgeUnlock[] = []; if (analysis.has_character && !badges.commander) unlocks.push({ id: 'commander', label: 'üéØ Commander' }); if (analysis.has_action && !badges.director) unlocks.push({ id: 'director', label: 'üé¨ Director' }); if (analysis.has_sequence && !badges.storyteller) unlocks.push({ id: 'storyteller', label: 'üìù Storyteller' }); if (analysis.has_detail && !badges.detail_master) unlocks.push({ id: 'detail_master', label: 'üé® Detail Master' }); if (analysis.has_environment && !badges.world_builder) unlocks.push({ id: 'world_builder', label: 'üåç World Builder' }); if (analysis.has_multi_char && !badges.combo_king) unlocks.push({ id: 'combo_king', label: 'üé™ Combo King' }); // Prompt Master: 4+ skills in one prompt const skillCount = Object.values(analysis).filter(Boolean).length; if (skillCount >= 4 && !badges.prompt_master) unlocks.push({ id: 'prompt_master', label: '‚≠ê Prompt Master' }); return unlocks; }
Persistence

Progress Tracking ‚Äî Keep It Simple

For a hackathon, you don't need a database. Here are three options in order of simplicity:

Option 1: localStorage Only Recommended for Hackathon

// Simple, zero-infra persistence interface PlayerProgress { username: string; badges: Record<string, boolean>; worlds_visited: string[]; total_prompts: number; best_prompt_skills: number; // highest skill count in a single prompt prompt_history: { text: string; world: string; timestamp: number; }[]; created_at: number; } // On app load const progress = JSON.parse(localStorage.getItem('pq_progress')) || createNewPlayer(); // On name entry (first launch) function createNewPlayer(name: string): PlayerProgress { return { username: name, badges: {}, worlds_visited: [], total_prompts: 0, best_prompt_skills: 0, prompt_history: [], created_at: Date.now() }; } // After each prompt function saveProgress(progress: PlayerProgress) { localStorage.setItem('pq_progress', JSON.stringify(progress)); }
Why this is enough: The demo is per-device. A kid at home will always use the same tablet/computer. localStorage persists indefinitely until cleared. For a hackathon demo, this is plenty ‚Äî judges don't care about cross-device sync. If you want to show "returning player" in the demo, just pre-seed localStorage before recording.

Option 2: Anthropic Artifact Storage API If Building as Artifact

If you deploy as a Claude artifact (React), you can use window.storage for persistence that survives sessions. Same data model as above, but replace localStorage calls with:

await window.storage.set('player:drew-jr', JSON.stringify(progress)); const result = await window.storage.get('player:drew-jr'); const progress = result ? JSON.parse(result.value) : createNewPlayer();

Option 3: Vercel KV / Upstash Redis Only If Multi-User Demo Needed

If you need to show multiple named players with a leaderboard (e.g., during a live demo), add a simple KV store. One serverless function:

// api/progress.ts (Vercel serverless) import { kv } from '@vercel/kv'; export async function GET(req) { const name = new URL(req.url).searchParams.get('name'); const data = await kv.get(`player:${name}`); return Response.json(data || null); } export async function POST(req) { const body = await req.json(); await kv.set(`player:${body.username}`, body); return Response.json({ ok: true }); }

Vercel KV has a free tier (256MB, 30K requests/month). More than enough for hackathon + demo.
